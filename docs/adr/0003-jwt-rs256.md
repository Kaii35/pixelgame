# ADR 0003 — JWT RS256 (asymmetric signing)

Status: Accepted
Date: 2026-05-15

## Context

Two services (API and Game Server) must agree on user identity. The API mints tokens at login. The Game Server validates them on `onAuth`. We need a trust mechanism between them.

Options:

- HS256 (shared symmetric secret): simpler, but the secret must be rotated in both services in lockstep, and any leak from either side compromises issuance.
- RS256 (asymmetric): only the API holds the private key; everyone else verifies with the public key. Compromise of a verifier doesn't enable token minting.

## Decision

Use **RS256** for access tokens.

- Keypair is generated locally via `scripts/gen-jwt-keys.mjs` into `./keys/`.
- API: `JWT_PRIVATE_KEY_PATH` + `JWT_PUBLIC_KEY_PATH`.
- Game Server: `JWT_PUBLIC_KEY_PATH` only.
- Tokens include `iss=pixelgame`, `aud=pixelgame-client`, short `exp` (15min default).
- Refresh tokens are opaque random strings, hashed (sha256) and stored in Postgres, with a `family_id` for rotation/reuse detection.

## Consequences

- Setup requires generating and distributing keys (handled by the script).
- Verifiers can be lightweight — no DB lookup per request.
- Rotation involves overlap: publish new public key, issue with new private key, retire old key after TTL.
